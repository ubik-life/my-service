## Решения

### 1. OpenAPI 3.0 вместо 3.1

Выбрали OpenAPI 3.0.3 — более широкая поддержка инструментами (генерация кода, валидация). Разница с 3.1 для нашего API незначительна.

### 2. POST вместо PUT для завершения фаз

Согласно `00-intent.md`, завершение регистрации и входа — POST к вложенным ресурсам:
- `POST /registrations/{id}/attestation`
- `POST /sessions/{id}/assertion`

Причина: POST семантически корректнее для продолжения процесса. PUT означает "положи ресурс целиком", что не отражает природу действия.

### 3. Формат base64url

Все бинарные данные (challenge, ключи, подписи) передаются в **base64url** (RFC 4648 §5). Это стандарт WebAuthn — URL-safe кодирование без padding.

### 4. Коды ошибок

Следуем принципу из `AGENTS.md` §1:
- `400` — невалидный JSON
- `409` — handle уже занят
- `422` — ошибка валидации (с детализацией по полям)
- `404` — challenge не найден / истёк
- `401` — подпись невалидна / токен невалидный

Единый формат error response:
```json
{
  "code": "VALIDATION_ERROR",
  "message": "...",
  "details": [{"field": "handle", "reason": "..."}]
}
```

### 5. JWT алгоритм Ed25519

Выбрали **Ed25519** (EdDSA) вместо RS256/ES256:
- Быстрее генерация и проверка
- Меньше размер ключей (32 байта vs 256 байт для RSA)
- Меньше размер подписи (64 байта vs 256 байт для RSA)
- Детерминированная подпись (в отличие от ECDSA)

### 6. Access token TTL 15 мин, refresh token TTL 30 дней

Компромисс между безопасностью и UX:
- **15 мин** — достаточно короткий TTL для access token (минимизация риска при утечке)
- **30 дней** — refresh token живёт долго, но инвалидируется при выходе

Access token не сохраняется в БД (stateless). Refresh token — opaque string, хранится в БД.

### 7. DELETE /sessions/current вместо /sessions/{id}

Упрощение для клиента: не нужно знать session_id. Сервер извлекает user_id из JWT и инвалидирует все refresh_tokens пользователя.

Примечание: access_token продолжит работать до истечения TTL. Полная инвалидация требует blacklist (не реализуем в demo).

## Альтернативы

### Async API (отвергли)

Рассматривали AsyncAPI 3.0 для будущего расширения (события регистрации, логины в Kafka).

Решение: пока не нужно. Сервис полностью синхронный. Добавим AsyncAPI при интеграции с event bus.

### Refresh token endpoint (отвергли)

Рассматривали `POST /tokens/refresh` для обновления access token.

Решение: не реализуем в demo. Клиент может использовать refresh token через custom механизм, но для простоты фокусируемся на WebAuthn-флоу.

### Handle validation — regex

Ограничение: `^[a-z0-9_-]+$`, длина 3-32.

Причина: совместимость с URL (handle может использоваться в будущих эндпоинтах типа `/users/{handle}`).
